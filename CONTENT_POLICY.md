# Content Policy

This policy governs non-code contributions and any mixed contributions that include creative, editorial, training, or publication assets.

## Purpose

Software licensing alone does not adequately protect a repository that accepts creative or publication material. Content can carry separate copyright, attribution, publicity, trademark, privacy, or redistribution restrictions.

For that reason, all content intended for the public BiblionOCR repository must be reviewed for provenance and publication rights.

## Covered Material

This policy applies to material such as:

- written content and publication copy
- screenshots and marketing graphics
- icons, illustrations, and photographs
- audio and video
- OCR reference material and training assets
- page images, transcription sets, and derived editorial resources

## Acceptance Standard

Content should be merged only when the project can clearly answer all of the following:

- Who created it?
- Does the contributor own it or have the right to submit it?
- Is public GitHub redistribution allowed?
- Are attribution or notice requirements known?
- Are there downstream restrictions that must be preserved?

If the answer to any of those questions is unclear, do not merge the asset into the public repository.

## Allowed Content Sources

Preferred sources:

- fully original content created by the contributor
- public-domain material with documented provenance
- third-party material under a license that clearly permits redistribution in this repository
- project-owned media with documented publication rights

## Restricted or Disallowed Content

Do not merge the following into the public repository unless the rights have been explicitly verified:

- stock assets with non-redistributable terms
- copyrighted scans, texts, or media without clear permission
- training data or reference sets with unclear provenance
- personal or sensitive information
- assets that impose obligations the project is not prepared to carry

## Contributor Statement

Anyone submitting covered content should be prepared to state, in the pull request or accompanying review notes:

- the origin of the material
- the license or permission basis
- whether attribution is required
- whether any file-level notice should be retained in the repository

## Repository Workflow Guidance

- Keep content review separate from normal code review when practical.
- Prefer pull requests over direct pushes.
- If contributors are primarily content creators rather than developers, prefer a curated intake path rather than broad write access to the main repository.
- If needed, use a separate private intake repository or off-repo review process for preliminary content screening.

## Relationship to Licensing

The root [LICENSE](LICENSE) covers original BiblionOCR software unless a file or directory states otherwise. That does not automatically determine whether contributed content is safe to publish.

See [THIRD_PARTY_LICENSES.md](THIRD_PARTY_LICENSES.md) for repository licensing notes and [CONTRIBUTING.md](CONTRIBUTING.md) for contributor workflow expectations.
